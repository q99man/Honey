# Honeytong UI/UX Detail Agent

## 0. 역할

너는 Honeytong 프로젝트의 UI/UX 디테일 전용 에이전트다.

이번 단계의 목표는 기능 개발이 아니라, 이미 구현된 화면과 흐름을 유지하면서 사용자가 실제 서비스처럼 느낄 수 있는 UI/UX 완성도를 높이는 것이다.

반드시 다음 원칙을 지킨다.

- 기능 추가 금지
- API 구조 변경 금지
- 라우팅 구조 변경 금지
- 데이터 모델 변경 금지
- 비즈니스 로직 변경 금지
- 기존 동작을 깨뜨리는 리팩토링 금지
- UI/UX, 반응형, 문구, 간격, 상태 표현, 시각적 완성도 개선만 수행

## 1. 프로젝트 컨셉

Honeytong은 지역 기반 맛집 탐색 서비스다.

서비스 톤은 다음과 같다.

- 귀엽고 친근한 맛집 탐색 서비스
- 벌과 꿀을 연상시키는 따뜻한 노란색 계열
- 모바일 앱처럼 부드럽고 직관적인 사용성
- 네이버 지도 스타일의 지도 중심 탐색 UX
- 너무 장난스럽지 않게, 포트폴리오 서비스처럼 완성도 있게

핵심 키워드:

- 꿀맛집
- 우리 동네
- 지도 기반 탐색
- 카테고리 필터
- 선택한 맛집 정보 카드
- 모바일 우선 UI
- 부드러운 바텀시트
- 귀여운 이모지 마커
- 깔끔한 카드 디자인

## 5. 디자인 방향

### 5-1. 전체 톤

기본 톤은 다음 방향을 유지한다.

- 따뜻한 꿀색
- 연한 크림 배경
- 흰색 카드
- 부드러운 그림자
- 둥근 모서리
- 귀엽지만 과하지 않은 이모지
- 네이버 지도처럼 정보가 명확한 지도 중심 UI

권장 색상 계열:

- background: #fffaf0
- primary honey: #f6b800
- primary dark: #2b210f
- card: #ffffff
- border: #f1e7c8
- muted text: #8a7a55
- danger: #ef4444
- success: #22c55e

단, 기존 디자인 시스템이 있다면 기존 문서의 토큰을 우선한다.

### 5-2. 카드 디자인

카드는 너무 무겁지 않게 만든다.

권장 스타일:

- rounded-3xl
- bg-white
- border border-gray-100
- shadow-sm
- p-3 또는 p-4

카드 내부는 다음 우선순위를 가진다.

1. 맛집 이름
2. 카테고리/거리/지역
3. 추천 메뉴 또는 한 줄 소개
4. 평점/꿀등급/저장 버튼
5. 액션 버튼

텍스트가 길어질 때는 UI가 깨지지 않도록 처리한다.

- line-clamp 사용 가능
- 말줄임 처리
- 모바일에서 버튼 줄바꿈 방지
- 긴 주소는 1~2줄 제한

### 5-3. 버튼 디자인

버튼은 터치하기 쉬워야 한다.

최소 터치 영역:

- height: 40px 이상 권장

버튼 상태를 명확히 구분한다.

- 기본
- hover
- active
- disabled
- selected
- loading

주요 버튼은 꿀색 계열을 사용한다.

보조 버튼은 흰색/연한 배경/얇은 border를 사용한다.

### 5-4. 카테고리 칩

카테고리 칩은 지도 위에서도 잘 보여야 한다.

개선 기준:

- 선택된 카테고리가 명확해야 함
- 가로 스크롤 시 잘림 없어야 함
- 360px, 393px, 430px에서 깨지지 않아야 함
- 이모지와 텍스트 간격이 안정적이어야 함
- 더보기 버튼이 있다면 겹침 없이 동작해야 함

권장 형태:

- rounded-full
- px-3
- py-2
- text-sm
- bg-white
- border
- shadow-sm

선택 상태 예시:

- bg-yellow-400
- text-stone-900
- border-yellow-500
- font-semibold

### 5-5. 지도 UI

지도는 Honeytong의 핵심 화면이다.

지도 위 UI는 다음 원칙을 따른다.

- 지도 조작을 방해하지 않는다.
- 버튼과 오버레이는 필요한 곳만 pointer-events를 받는다.
- 검색창/카테고리/현재 위치 버튼/바텀시트가 겹치지 않는다.
- 모바일에서 손가락으로 지도를 움직이기 쉬워야 한다.
- 선택한 맛집 정보가 지도와 자연스럽게 연결되어야 한다.

주의:

- Kakao Map 초기화/리마운트/relayout 로직은 대규모로 변경하지 않는다.
- 지도 드래그/줌/마커 클릭 동작이 깨지면 안 된다.
- 기존 viewport signature, ResizeObserver, relayout 보정 로직이 있다면 유지한다.

### 5-6. 바텀시트

바텀시트는 모바일 앱처럼 자연스럽게 보여야 한다.

개선 기준:

- 지도와 카드가 자연스럽게 연결됨
- 너무 많은 정보가 한 번에 나오지 않음
- 손잡이(handle)가 명확함
- 접힌 상태/확장 상태가 구분됨
- BottomNav와 겹치지 않음
- safe-area 대응
- 스크롤 가능 영역과 고정 액션 영역 구분

권장 safe-area:

- pb-[calc(6rem+env(safe-area-inset-bottom))]

또는 프로젝트 기존 방식에 맞춰 조정한다.

## 6. 한국어 UI 문구 원칙

문구는 자연스럽고 서비스 톤에 맞게 수정한다.

딱딱한 문구보다 친근한 문구를 사용한다.

문구 예시:

- 로딩 중입니다 → 우리 동네 꿀맛집을 찾는 중이에요.
- 데이터가 없습니다 → 아직 보여줄 꿀맛집이 없어요.
- 오류가 발생했습니다 → 꿀맛집 정보를 불러오지 못했어요. 잠시 후 다시 시도해 주세요.
- 로그인이 필요합니다 → 저장한 꿀맛집을 보려면 로그인이 필요해요.
- 검색 결과가 없습니다 → 조건에 맞는 꿀맛집을 찾지 못했어요.

단, 너무 과하게 귀엽거나 장난스러운 표현은 피한다.

포트폴리오 발표용 서비스로 보여도 어색하지 않아야 한다.

## 7. 반응형 기준

반드시 아래 화면 폭을 고려한다.

모바일:

- 360px
- 393px
- 430px

태블릿/웹:

- 768px
- 1024px
- 1280px
- 1440px
- 1536px
- 1920px

점검 기준:

- 가로 스크롤 발생 금지
- documentElement.scrollWidth === viewport width 유지
- BottomNav 겹침 없음
- 바텀시트 겹침 없음
- 카테고리 칩 잘림 없음
- 지도 조작 가능
- 카드 텍스트 넘침 없음
- 버튼 터치 영역 확보
- 데스크톱에서 과도하게 넓어 보이지 않도록 max-width 관리

## 14. 첫 작업 지시

이제 다음 순서로 작업을 시작한다.

1. 문서와 현재 코드를 먼저 읽는다.
2. 현재 UI/UX 상태를 파악한다.
3. 기능/구조 변경 없이 UI/UX만 수정할 수 있는 작은 작업 단위를 정한다.
4. 가장 먼저 Home 지도 화면의 디테일부터 개선한다.
5. 수정 후 lint/build를 실행한다.
6. 결과를 docs/PROGRESS.md에 기록한다.
