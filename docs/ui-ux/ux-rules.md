# Honeytong UX Rules

## 1. 문서 목적

이 문서는 Honeytong 프로젝트의 사용자 경험 기준을 정의한다.

Honeytong은 벌과 꿀을 테마로 한 동네 맛집 탐색 웹앱이다.  
사용자는 앱을 통해 우리 동네의 맛집을 탐색하고, 찜하고, 랭킹을 확인하고, 자신의 활동을 관리할 수 있다.

이 문서는 Codex가 UI/UX 리팩토링을 진행할 때 화면 구조, 정보 배치, 사용자 흐름, 빈 상태, 로딩 상태, 에러 상태, 문구 톤을 일관되게 유지하도록 하기 위한 기준이다.

---

## 2. UX 핵심 목표

Honeytong의 UX 목표는 다음과 같다.

1. 사용자가 앱에 들어오자마자 동네 맛집 탐색 앱이라는 것을 이해한다.
2. 맛집 정보를 카드 중심으로 쉽고 빠르게 훑어볼 수 있다.
3. 마음에 드는 맛집을 자연스럽게 찜할 수 있다.
4. 상세 화면에서 방문 여부를 판단할 수 있다.
5. 랭킹 화면에서 우리 동네 인기 맛집을 재미있게 확인할 수 있다.
6. 마이페이지에서 내 활동과 동네 인증 상태를 확인할 수 있다.
7. 전체적으로 따뜻하고 친근한 Honeytong 감성을 느낄 수 있다.

---

## 3. UX 기본 원칙

### 3.1 모바일 우선

Honeytong은 모바일 사용성을 가장 우선한다.

기준 화면은 다음과 같다.

```txt
393px x 852px
```

모든 화면은 먼저 모바일 기준으로 보기 좋아야 한다.  
데스크톱에서는 모바일 앱이 중앙에 놓인 것처럼 보이게 한다.

권장 최대 폭:

```txt
430px
```

---

### 3.2 한 화면, 한 목적

각 화면은 하나의 주요 목적을 가져야 한다.

| 화면 | 주요 목적 |
|---|---|
| Home | 동네 맛집 탐색 |
| Detail | 맛집 정보 확인 |
| Ranking | 지역별 인기 맛집 확인 |
| Wishlist | 찜한 맛집 관리 |
| MyPage | 내 활동과 계정 상태 확인 |

Codex는 한 화면에 너무 많은 기능을 몰아넣지 않는다.

---

### 3.3 사용자가 길을 잃지 않게 하기

사용자는 항상 자신이 어느 화면에 있는지 알 수 있어야 한다.

이를 위해 다음 요소를 유지한다.

- 화면 제목
- 하단 네비게이션 활성 상태
- 상세 화면의 뒤로가기 버튼
- 섹션 제목
- 선택된 필터 표시
- 현재 지역 또는 카테고리 표시

---

### 3.4 정보는 단계적으로 보여주기

목록 카드에서는 핵심 정보만 보여준다.

카드에서 보여줄 정보:

- 맛집 이름
- 대표 이미지
- 지역
- 거리
- 카테고리
- 한 줄 설명
- 추천 수 또는 꿀 점수
- 찜 상태

상세 화면에서 보여줄 정보:

- 상세 설명
- 운영 정보
- 위치 정보
- 추천 이유
- 방문 인증 정보
- 랭킹 정보
- 리뷰 또는 댓글

목록 카드에 모든 정보를 넣으려고 하지 않는다.

---

### 3.5 사용자의 다음 행동을 명확하게 만들기

사용자는 각 화면에서 다음에 무엇을 할 수 있는지 쉽게 알아야 한다.

예시:

- 홈에서는 맛집 카드를 눌러 상세로 이동할 수 있어야 한다.
- 카드에서는 찜 버튼을 누를 수 있어야 한다.
- 상세에서는 찜하기, 추천하기, 방문 인증하기, 길찾기 같은 행동이 보여야 한다.
- 찜 목록이 비어 있으면 맛집 둘러보기 버튼이 있어야 한다.
- 에러가 발생하면 다시 불러오기 버튼이 있어야 한다.

## 4. Home 화면 UX 규칙

### 4.1 Home 화면 목적

Home 화면은 Honeytong의 첫인상을 결정하는 화면이다.

사용자는 Home 화면에서 다음을 바로 이해해야 한다.

```txt
우리 동네의 꿀맛집을 볼 수 있다.
동네 사람들이 추천한 맛집을 찾을 수 있다.
마음에 드는 맛집을 찜할 수 있다.
```

---

### 4.2 Home 화면 정보 구조

Home 화면은 다음 순서로 구성한다.

1. 상단 브랜드/인사 영역
2. 검색창 또는 지역 안내 영역
3. 카테고리 필터 칩
4. 추천 맛집 섹션
5. 인기 상승 맛집 또는 랭킹 미리보기
6. 하단 네비게이션

---

### 4.3 Home 상단 문구

Home 화면 상단은 Honeytong의 정체성을 보여줘야 한다.

권장 문구:

```txt
오늘은 어디서 꿀맛을 찾을까요?
우리 동네 사람들이 추천한 맛집을 모아봤어요.
```

또는:

```txt
우리 동네 꿀맛집을 찾아볼까요?
동네 사람들이 자주 찾는 맛집을 소개해드려요.
```

피해야 할 문구:

```txt
Home
Main Page
Restaurant List
Welcome User
```

---

### 4.4 Home 검색 UX

검색창은 사용자가 맛집, 메뉴, 지역을 찾을 수 있다는 느낌을 준다.

placeholder 예시:

```txt
동네 맛집이나 메뉴를 검색해보세요
```

검색 기능이 아직 구현되지 않았다면 다음 중 하나로 처리한다.

1. 비활성 스타일로 표시한다.
2. 클릭 시 “검색 기능은 곧 추가될 예정이에요.” 안내를 보여준다.
3. 실제 기능이 아닌데 작동하는 것처럼 보이게 만들지 않는다.

---

### 4.5 Home 카테고리 UX

카테고리는 가로 스크롤 칩 형태를 권장한다.

기본 카테고리 예시:

```txt
전체 / 한식 / 분식 / 카페 / 디저트 / 중식 / 일식 / 양식 / 혼밥 / 가성비
```

규칙:

- 선택된 카테고리는 꿀색 배경으로 강조한다.
- 선택되지 않은 카테고리는 흰색 또는 연한 회색 배경으로 표시한다.
- 카테고리 선택 후 결과가 없다면 EmptyState를 보여준다.
- 카테고리 칩은 터치하기 충분한 크기를 가진다.

---

### 4.6 Home 카드 리스트 UX

맛집 카드는 Home 화면의 핵심이다.

카드 UX 규칙:

- 카드 전체를 누르면 상세 화면으로 이동한다.
- 찜 버튼을 누르면 찜 상태만 바뀌어야 한다.
- 찜 버튼 클릭 시 상세 화면으로 이동하면 안 된다.
- 이미지, 이름, 설명, 거리, 추천 수가 명확히 보여야 한다.
- 카드 간격은 답답하지 않게 유지한다.
- 카드가 너무 복잡해 보이면 정보를 줄인다.

---

### 4.7 Home 상태 처리

Home 화면은 다음 상태를 반드시 고려한다.

#### Loading

```txt
꿀맛집을 불러오는 중이에요...
```

#### Empty

```txt
아직 보여줄 맛집이 없어요.
조금 뒤에 다시 확인해보세요.
```

#### Error

```txt
맛집 정보를 불러오지 못했어요.
잠시 후 다시 시도해주세요.
```

버튼:

```txt
다시 불러오기
```

---

## 5. Detail 화면 UX 규칙

### 5.1 Detail 화면 목적

Detail 화면은 사용자가 해당 맛집에 실제로 관심을 가질지 판단하는 화면이다.

사용자는 Detail 화면에서 다음을 확인할 수 있어야 한다.

- 어떤 맛집인지
- 어디에 있는지
- 어떤 카테고리인지
- 왜 추천받는지
- 얼마나 인기 있는지
- 찜하거나 추천할 수 있는지

---

### 5.2 Detail 화면 정보 구조

Detail 화면은 다음 순서로 구성한다.

1. 상단 대표 이미지
2. 뒤로가기 버튼
3. 찜 버튼
4. 맛집 이름
5. 지역, 거리, 카테고리
6. 꿀 점수 또는 추천 수
7. 상세 설명
8. 운영 정보
9. 위치 정보
10. 추천하기 또는 방문 인증 CTA

---

### 5.3 Detail 상단 이미지 UX

상세 화면의 대표 이미지는 맛집의 첫인상을 결정한다.

규칙:

- 화면 상단에 크게 보여준다.
- 이미지 비율은 안정적으로 유지한다.
- 이미지가 없을 때 fallback을 제공한다.
- 뒤로가기 버튼과 찜 버튼은 이미지 위에 둘 수 있다.
- 이미지 위에 텍스트를 과하게 올리지 않는다.

---

### 5.4 Detail 핵심 정보 UX

이미지 아래에는 핵심 정보를 바로 보여준다.

예시:

```txt
성수 꿀분식
성수동 · 분식 · 650m
동네 사람들이 자주 추천한 꿀맛집이에요.
```

규칙:

- 맛집 이름은 가장 크게 보여준다.
- 지역, 거리, 카테고리는 한 줄로 정리한다.
- 설명은 너무 길지 않게 시작한다.
- 긴 설명은 상세 섹션에서 보여준다.

---

### 5.5 Detail CTA UX

상세 화면의 CTA는 실제 구현된 기능만 활성화한다.

가능한 CTA:

- 찜하기
- 추천하기
- 방문 인증하기
- 길찾기

아직 구현되지 않은 기능은 다음과 같이 처리한다.

```txt
곧 추가될 기능이에요.
```

또는 비활성 버튼으로 표시한다.

---

### 5.6 Detail 상태 처리

#### Loading

```txt
맛집 정보를 불러오는 중이에요...
```

#### Error

```txt
맛집 정보를 불러오지 못했어요.
잠시 후 다시 시도해주세요.
```

#### Empty 또는 Not Found

```txt
해당 맛집을 찾을 수 없어요.
```

버튼:

```txt
홈으로 돌아가기
```

## 6. Ranking 화면 UX 규칙

### 6.1 Ranking 화면 목적

Ranking 화면은 Honeytong의 차별성을 보여주는 화면이다.

사용자는 Ranking 화면에서 다음을 확인할 수 있어야 한다.

- 우리 동네에서 인기 있는 맛집
- 구 단위로 인기 있는 맛집
- 시 단위로 인기 있는 맛집
- 최근 추천이 많은 맛집
- 방문 활동이 많은 맛집

---

### 6.2 Ranking 화면 정보 구조

Ranking 화면은 다음 순서로 구성한다.

1. 화면 제목
2. 랭킹 설명 문구
3. 지역 단위 필터
4. Top 3 강조 영역
5. 전체 랭킹 리스트
6. 하단 네비게이션

---

### 6.3 Ranking 필터 UX

지역 단위 필터 예시:

```txt
동네 / 구 / 시
```

규칙:

- 선택된 필터는 꿀색으로 강조한다.
- 필터를 바꾸면 랭킹 목록도 바뀌어야 한다.
- API 연동 전이라면 실제 동작 가능한 범위만 보여준다.
- 필터가 작동하지 않는다면 준비 중 안내가 필요하다.

---

### 6.4 Ranking 카드 UX

랭킹 카드는 순위가 명확하게 보여야 한다.

카드에 포함할 정보:

- 순위
- 맛집 이미지
- 맛집 이름
- 지역
- 추천 수
- 방문 수
- 짧은 추천 이유

Top 3는 약간 더 강조할 수 있다.

주의:

- 너무 게임 랭킹처럼 과장하지 않는다.
- 왕관, 꿀단지, 벌 아이콘은 적당히 사용한다.
- 순위 숫자보다 맛집 정보가 더 잘 보여야 한다.

---

### 6.5 Ranking 기준 문구

랭킹 기준은 사용자에게 짧게 알려준다.

예시:

```txt
최근 추천, 찜, 방문 활동을 바탕으로 정리했어요.
```

또는:

```txt
동네 사람들의 활동을 기준으로 인기 맛집을 보여드려요.
```

---

### 6.6 Ranking 상태 처리

#### Loading

```txt
랭킹을 불러오는 중이에요...
```

#### Empty

```txt
아직 랭킹에 올라온 맛집이 없어요.
첫 번째 꿀맛집을 추천해보세요.
```

#### Error

```txt
랭킹 정보를 불러오지 못했어요.
잠시 후 다시 시도해주세요.
```

---

## 7. Wishlist 화면 UX 규칙

### 7.1 Wishlist 화면 목적

Wishlist 화면은 사용자가 찜한 맛집을 다시 확인하고 관리하는 화면이다.

사용자는 Wishlist 화면에서 다음을 할 수 있어야 한다.

- 내가 찜한 맛집 목록 보기
- 찜한 맛집 상세로 이동하기
- 찜 해제하기
- 찜 목록이 비었을 때 맛집 탐색으로 이동하기

---

### 7.2 Wishlist 화면 정보 구조

Wishlist 화면은 다음 순서로 구성한다.

1. 화면 제목
2. 찜 개수 요약
3. 찜한 맛집 리스트
4. 빈 상태
5. 하단 네비게이션

---

### 7.3 Wishlist 빈 상태 UX

찜 목록이 비었을 때는 친근하게 안내한다.

문구:

```txt
아직 찜한 맛집이 없어요.
마음에 드는 맛집을 꿀단지에 담아보세요.
```

버튼:

```txt
맛집 둘러보기
```

---

### 7.4 Wishlist 카드 UX

Wishlist의 카드는 Home의 RestaurantCard와 최대한 일관되게 유지한다.

규칙:

- 같은 카드 컴포넌트를 재사용하는 것을 우선한다.
- 찜 해제 버튼은 명확해야 한다.
- 찜 해제 후 목록에서 제거되거나 상태가 즉시 반영되어야 한다.
- API 실패 시 사용자에게 안내한다.

---

### 7.5 Wishlist 상태 처리

#### Loading

```txt
찜한 맛집을 불러오는 중이에요...
```

#### Empty

```txt
아직 찜한 맛집이 없어요.
마음에 드는 맛집을 꿀단지에 담아보세요.
```

##### Error

```txt
찜 목록을 불러오지 못했어요.
잠시 후 다시 시도해주세요.
```

---

## 8. MyPage 화면 UX 규칙

### 8.1 MyPage 화면 목적

MyPage 화면은 사용자의 활동과 계정 상태를 보여주는 화면이다.

사용자는 MyPage에서 다음을 확인할 수 있어야 한다.

- 내 프로필
- 동네 인증 상태
- 내가 찜한 맛집 수
- 내가 추천한 맛집 수
- 방문 인증 수
- 설정 또는 로그아웃

---

### 8.2 MyPage 화면 정보 구조

MyPage 화면은 다음 순서로 구성한다.

1. 프로필 요약 카드
2. 동네 인증 상태

### 8.3 MyPage 프로필 UX

프로필 영역은 Honeytong 감성을 살릴 수 있는 곳이다.

예시 문구:

```txt
꿀벌 성오님
우리 동네 꿀맛집을 찾는 중이에요.
```

또는:

```txt
오늘도 꿀맛집 탐색 중이에요.
```

규칙:

- 너무 과하게 게임 캐릭터처럼 만들지 않는다.
- 프로필 정보가 없을 때 fallback을 제공한다.
- 동네 인증 상태는 명확히 표시한다.

### 8.4 MyPage 활동 요약 UX

활동 요약은 카드 형태로 보여준다.

예시 항목:

```txt
찜한 맛집
추천한 맛집
방문 인증
꿀 포인트
```

규칙:

- 숫자는 크게, 설명은 작게 표시한다.
- 없는 데이터는 0으로 표시한다.
- 추후 API 확장 가능성을 고려한다.

## 9. 공통 상태 UX

### 9.1 Loading State

로딩 중에는 화면이 완전히 비어 보이면 안 된다.

권장 방식:

- Skeleton 카드
- 간단한 로딩 문구
- 버튼 내부 로딩 상태

권장 문구:

```txt
꿀맛집을 불러오는 중이에요...
```

피해야 할 문구:

```txt
Loading...
```

---

### 9.2 Empty State

빈 상태는 다음 행동을 안내해야 한다.

예시:

```txt
아직 보여줄 맛집이 없어요.
새로운 꿀맛집이 등록되면 알려드릴게요.
```

또는:

```txt
아직 찜한 맛집이 없어요.
마음에 드는 맛집을 꿀단지에 담아보세요.
```

빈 상태에는 가능하면 CTA 버튼을 제공한다.

---

### 9.3 Error State

에러는 사용자 탓처럼 보이면 안 된다.

권장 문구:

```txt
정보를 불러오지 못했어요.
잠시 후 다시 시도해주세요.
```

또는 화면별로 구체화한다.

```txt
맛집 정보를 불러오지 못했어요.
랭킹 정보를 불러오지 못했어요.
찜 목록을 불러오지 못했어요.
```

버튼:

```txt
다시 불러오기
```

피해야 할 문구:

```txt
Error
Fail
Request Failed
Unknown Error
```

## 10. 문구 톤앤매너

Honeytong의 문구는 따뜻하고 친근해야 한다.

### 10.1 권장 톤

- 친근함
- 부드러움
- 동네 느낌
- 가볍고 귀여움
- 과하지 않은 유머

---

### 10.2 권장 문구

```txt
오늘은 어디서 꿀맛을 찾을까요?
우리 동네 꿀맛집을 모아봤어요.
동네 사람들이 자주 찾는 곳이에요.
마음에 드는 맛집을 꿀단지에 담아보세요.
아직 찜한 맛집이 없어요.
랭킹을 불러오는 중이에요.
잠시 후 다시 시도해주세요.
```

---

### 10.3 피해야 할 문구

```txt
Home
Main
Submit
Cancel
No Data
Error
Loading

Restaurant Detail

```

## 11. 사용자 흐름 기준

### 11.1 기본 탐색 흐름

기본 흐름은 다음과 같다.

```txt
Home
→ 맛집 카드 클릭
→ Detail
→ 찜 또는 추천
→ 뒤로가기
→ Home 또는 Ranking
```

---

### 11.2 찜 흐름

```txt
Home 또는 Detail
→ 찜 버튼 클릭
→ 찜 상태 변경
→ Wishlist에서 확인
```

주의:

- 찜 버튼 클릭 시 상세 이동이 발생하면 안 된다.
- API 실패 시 상태를 복구하거나 안내해야 한다.

---

### 11.3 랭킹 흐름

```txt
Ranking
→ 지역 단위 선택
→ 랭킹 카드 확인
→ 맛집 상세 이동
```

---

### 11.4 마이페이지 흐름

```txt
MyPage
→ 내 활동 확인
→ 찜 목록 이동
→ 설정 또는 로그아웃
```

## 12. UX 금지 사항

Codex는 UI/UX 리팩토링 중 다음을 하면 안 된다.

- 기존 기능 삭제
- 라우팅 구조 임의 변경
- API 연결 구조 변경
- mock 데이터 재도입
- 버튼을 눌렀는데 아무 반응 없게 만들기
- 클릭 가능한 것처럼 보이지만 클릭되지 않는 UI 만들기
- 상세 화면에 뒤로가기 없이 진입하게 만들기
- 카드 클릭과 찜 클릭이 충돌하게 만들기
- 영어 UI 문구로 바꾸기
- 화면 전체를 한 번에 갈아엎기
- 너무 많은 장식 요소 추가
- 모바일에서 터치하기 어려운 작은 버튼 만들기
- 빈 상태나 에러 상태를 흰 화면으로 방치하기

---

## 13. UX 리팩토링 완료 기준

Codex가 UX 개선을 완료했다면 다음을 확인해야 한다.

- 사용자가 현재 화면의 목적을 바로 이해할 수 있는가?
- 모바일 393x852 기준에서 보기 좋은가?
- 하단 네비게이션이 본문을 가리지 않는가?
- 카드 클릭과 찜 버튼 클릭이 충돌하지 않는가?
- 로딩, 빈 상태, 에러 상태가 처리되어 있는가?
- 문구가 한국어로 자연스러운가?
- 기존 API 연결이 유지되는가?
- 기존 라우팅이 유지되는가?
- TypeScript 오류가 없는가?
- 한 번에 너무 큰 범위를 수정하지 않았는가?
