# Honeytong Interaction Rules

## 1. 문서 목적

이 문서는 Honeytong 프로젝트의 사용자 상호작용 기준을 정의한다.

Honeytong은 벌과 꿀을 테마로 한 동네 맛집 탐색 웹앱이다.  
이 문서는 Codex가 버튼, 카드, 찜, 검색, 필터, 라우팅, 로딩, 에러, 빈 상태, 애니메이션을 수정할 때 기존 기능을 깨뜨리지 않고 일관된 UX를 유지하도록 하기 위한 기준이다.

Codex는 UI/UX 리팩토링 중 기능 동작을 임의로 바꾸지 말고, 이 문서의 인터랙션 규칙을 따라야 한다.

---

## 2. 기본 인터랙션 원칙

Honeytong의 모든 상호작용은 다음 원칙을 따른다.

1. 클릭 가능한 요소는 클릭 가능해 보여야 한다.
2. 클릭 후 사용자가 변화를 느낄 수 있어야 한다.
3. 버튼과 카드의 터치 영역은 충분히 커야 한다.
4. 카드 클릭과 내부 버튼 클릭은 충돌하지 않아야 한다.
5. 로딩, 에러, 빈 상태는 반드시 사용자에게 안내해야 한다.
6. 미구현 기능은 진짜 기능처럼 보이면 안 된다.
7. 애니메이션은 짧고 부드럽게 사용한다.
8. 모바일 사용성을 우선한다.
9. 한국어 UI 문구를 유지한다.
10. 기존 API, 라우팅, 상태 관리를 깨뜨리지 않는다.

---

## 3. 터치 영역 규칙

### 3.1 최소 터치 영역

모바일 기준으로 버튼과 아이콘 버튼은 충분한 터치 영역을 가져야 한다.

권장 최소 크기:

```txt
44px x 44px
```

아이콘 자체가 작더라도 버튼 영역은 충분히 커야 한다.

권장 스타일:

```tsx
className="flex h-11 w-11 items-center justify-center rounded-full"
```

또는 최소:

```tsx
className="flex h-10 w-10 items-center justify-center rounded-full"
```

---

### 3.2 터치 영역 금지 사항

다음은 피한다.

- 아이콘 크기만 버튼 크기로 사용하는 것
- 24px 이하의 작은 클릭 영역
- 텍스트만 좁게 눌리는 버튼
- 카드 안의 여러 버튼이 너무 가까운 배치
- 하단 네비게이션 메뉴가 너무 좁은 배치

---

## 4. 카드 클릭 규칙

### 4.1 RestaurantCard 클릭

맛집 카드는 클릭 시 상세 화면으로 이동한다.

예시:

```tsx
onClick={() => navigate(`/restaurants/${id}`)}
```

단, 실제 프로젝트의 라우팅 경로가 다르다면 기존 경로를 유지한다.

Codex는 라우팅 경로를 임의로 바꾸면 안 된다.

---

### 4.2 카드 내부 버튼 클릭

카드 내부에는 찜 버튼, 공유 버튼, 더보기 버튼 등이 있을 수 있다.  
이 버튼들은 카드 전체 클릭과 충돌하면 안 된다.

찜 버튼 예시:

```tsx
<button
  type="button"
  aria-label={isWished ? "찜 해제" : "찜하기"}
  onClick={(event) => {
    event.stopPropagation();
    onToggleWish?.();
  }}
>
  ...
</button>
```

---

### 4.3 카드 클릭 금지 사항

다음은 금지한다.

- 찜 버튼을 눌렀는데 상세 화면으로 이동하는 것
- 카드 내부 버튼을 눌렀는데 카드 onClick이 함께 실행되는 것
- 카드가 클릭 가능해 보이는데 아무 동작이 없는 것
- 카드 클릭 경로를 리팩토링 중 임의 변경하는 것
- 카드 전체에 너무 많은 버튼을 넣어 사용자가 헷갈리게 만드는 것

---

## 5. 찜 버튼 규칙

### 5.1 찜 버튼 목적

찜 버튼은 사용자가 관심 있는 맛집을 저장하거나 해제하는 핵심 인터랙션이다.

사용 위치:

- Home 맛집 카드
- Detail 화면
- Wishlist 화면
- Ranking 카드 일부

---

### 5.2 찜 상태

찜 버튼은 두 가지 상태를 가진다.

```txt
찜 안 됨
찜 됨
```

찜 안 됨 상태:

- 빈 하트 또는 기본 아이콘
- 흰색 또는 연한 배경
- 회색/진한 갈색 아이콘

찜 됨 상태:

- 채워진 하트 또는 강조 아이콘
- 꿀색 또는 따뜻한 강조색
- 명확하게 상태가 구분되어야 함

---

### 5.3 찜 클릭 동작

찜 버튼 클릭 시 다음 순서를 권장한다.

1. 사용자가 찜 버튼 클릭
2. UI 상태 즉시 변경
3. API 요청 실행
4. 성공 시 상태 유지
5. 실패 시 상태 복구 또는 에러 안내

실패 문구 예시:

```txt
찜 상태를 변경하지 못했어요.
잠시 후 다시 시도해주세요.
```

---

### 5.4 찜 버튼 이벤트 전파 방지

카드 내부에서 찜 버튼을 사용할 때는 반드시 이벤트 전파를 막는다.

```tsx
onClick={(event) => {
  event.stopPropagation();
  onToggleWish?.();
}}
```

---

### 5.5 찜 버튼 접근성

아이콘만 있는 찜 버튼은 `aria-label`을 제공한다.

```tsx
aria-label={isWished ? "찜 해제" : "찜하기"}
```

---

### 5.6 찜 버튼 금지 사항

다음은 금지한다.

- 찜 버튼 클릭 시 상세 화면으로 이동
- 찜 상태가 시각적으로 구분되지 않음
- API 실패 시 성공한 것처럼 계속 표시
- 찜 클릭 후 아무 피드백 없음
- 아이콘만 있고 label 또는 aria-label이 없음
- 너무 작은 찜 버튼

---

## 6. 하단 네비게이션 규칙

### 6.1 하단 네비게이션 목적

하단 네비게이션은 주요 화면 간 이동을 담당한다.

기본 메뉴:

```txt
홈
랭킹
찜
마이
```

---

### 6.2 활성 상태

현재 페이지에 해당하는 메뉴는 명확히 강조한다.

활성 메뉴 권장 스타일:

```tsx
className="text-[#d99a00]"
```

비활성 메뉴 권장 스타일:

```tsx
className="text-gray-400"
```

---

### 6.3 네비게이션 위치

하단 네비게이션은 모바일 앱처럼 하단에 고정한다.

권장 스타일:

```tsx
className="fixed bottom-0 left-1/2 z-50 w-full max-w-[430px] -translate-x-1/2 border-t border-gray-200 bg-white px-4 py-2"
```

본문이 가려지지 않도록 페이지에는 하단 여백을 둔다.

```tsx
className="pb-24"
```

---

### 6.4 하단 네비게이션 금지 사항

다음은 금지한다.

- 하단 네비게이션이 본문을 가리는 것
- 현재 메뉴 활성 상태가 없는 것
- 라우팅 경로를 임의로 변경하는 것
- 메뉴 텍스트를 영어로 변경하는 것
- 메뉴 클릭 시 아무 반응이 없는 것
- 데스크톱에서 전체 화면 폭으로 너무 넓게 퍼지는 것

---

## 7. 필터 / 카테고리 칩 규칙

### 7.1 필터 목적

필터와 카테고리 칩은 사용자가 원하는 맛집 목록을 빠르게 좁히기 위한 요소다.

사용 위치:

- Home 카테고리
- Ranking 지역 단위
- 검색 결과
- 정렬 옵션

---

### 7.2 선택 상태

선택된 칩은 꿀색 계열로 명확히 강조한다.

선택 전:

```tsx
className="rounded-full border border-gray-200 bg-white px-4 py-2 text-sm font-medium text-gray-600"
```

선택 후:

```tsx
className="rounded-full bg-[#f6b800] px-4 py-2 text-sm font-semibold text-[#2b210f]"
```

---

### 7.3 필터 동작

필터 클릭 시 다음 중 하나가 일어나야 한다.

- 목록이 즉시 변경된다.
- API 요청 후 목록이 변경된다.
- 미구현 상태라면 준비 중 안내를 보여준다.

준비 중 문구:

```txt
이 필터는 곧 추가될 예정이에요.
```

---

### 7.4 필터 결과 없음

필터 결과가 없으면 EmptyState를 보여준다.

예시:

```txt
조건에 맞는 맛집이 없어요.
다른 카테고리를 선택해보세요.
```

---

### 7.5 필터 금지 사항

다음은 금지한다.

- 선택된 칩이 구분되지 않는 것
- 필터 클릭 후 아무 변화가 없는 것
- 미구현 필터를 정상 기능처럼 보이게 하는 것
- 필터 클릭 시 기존 API 연결이 깨지는 것
- 칩 터치 영역이 너무 작은 것

---

## 8. 검색 인터랙션 규칙

### 8.1 검색 목적

검색은 사용자가 맛집, 메뉴, 지역을 직접 찾을 수 있게 하는 진입점이다.

검색 placeholder 예시:

```txt
동네 맛집이나 메뉴를 검색해보세요
```

---

### 8.2 검색 기능이 구현된 경우

검색 기능이 실제로 구현되어 있다면 다음을 지킨다.

- 입력값에 따라 검색 결과를 보여준다.
- 빈 검색어는 전체 목록 또는 기본 목록으로 돌아간다.
- 검색 결과가 없으면 EmptyState를 보여준다.
- 검색 중에는 loading 상태를 보여준다.
- 검색 실패 시 error 상태를 보여준다.

---

### 8.3 검색 기능이 미구현인 경우

검색 기능이 아직 구현되지 않았다면 활성 기능처럼 보이게 만들지 않는다.

가능한 처리:

1. 비활성 스타일
2. 클릭 시 준비 중 안내
3. 검색창 대신 지역 안내 카드 사용

준비 중 문구:

```txt
검색 기능은 곧 추가될 예정이에요.
```

---

### 8.4 검색 금지 사항

다음은 금지한다.

- 검색창을 눌렀는데 아무 반응이 없음
- 실제 검색이 안 되는데 검색 결과가 나오는 것처럼 보임
- 검색 실패 시 빈 화면만 보여줌
- placeholder만 영어로 작성
- 검색 입력 중 레이아웃이 깨짐

---

## 9. 버튼 인터랙션 규칙

### 9.1 버튼 종류

Honeytong에서 사용하는 버튼은 크게 다음과 같다.

```txt
Primary Button
Secondary Button
Ghost Button
Icon Button
CTA Button
```

---

### 9.2 Primary Button

주요 행동에 사용한다.

예시:

```txt
맛집 둘러보기
추천하기
방문 인증하기
저장하기
다시 불러오기
```

권장 스타일:

```tsx
className="rounded-full bg-[#f6b800] px-5 py-3 text-sm font-semibold text-[#2b210f] transition duration-200 active:scale-[0.98]"
```

---

### 9.3 Secondary Button

보조 행동에 사용한다.

예시:

```txt
취소
나중에 하기
필터
정렬
```

권장 스타일:

```tsx
className="rounded-full border border-gray-200 bg-white px-4 py-2 text-sm font-semibold text-gray-700"
```

---

### 9.4 Disabled Button

비활성 버튼은 명확히 비활성처럼 보여야 한다.

권장:

```tsx
disabled:opacity-50
disabled:cursor-not-allowed
```

비활성 버튼 클릭 시 실제 동작이 발생하면 안 된다.

---

### 9.5 Loading Button

버튼이 API 요청 중이라면 중복 클릭을 막는다.

예시 문구:

```txt
처리 중...
저장 중...
불러오는 중...
```

---

### 9.6 버튼 금지 사항

다음은 금지한다.

- 버튼 클릭 후 아무 변화 없음
- 비활성처럼 보이는데 실제 동작함
- 활성처럼 보이는데 클릭 불가
- 버튼 문구가 영어로 바뀜
- 너무 작은 버튼
- 미구현 기능 버튼을 활성 CTA처럼 표시

---

## 10. 로딩 상태 규칙

### 10.1 로딩 목적

로딩 상태는 API 요청 중 사용자가 기다릴 수 있도록 안내하는 역할을 한다.

---

### 10.2 권장 로딩 방식

Honeytong에서는 다음 방식을 권장한다.

- Skeleton 카드
- 짧은 한국어 문구
- 버튼 내부 loading 텍스트
- 너무 큰 spinner 남발 금지

권장 문구:

```txt
꿀맛집을 불러오는 중이에요...
```

화면별 문구:

```txt
맛집 정보를 불러오는 중이에요...
랭킹을 불러오는 중이에요...
찜한 맛집을 불러오는 중이에요...
내 정보를 불러오는 중이에요...
```

---

### 10.3 Skeleton 규칙

카드 목록에서는 SkeletonCard를 사용할 수 있다.

권장 개수:

```txt
3~5개
```

너무 많은 skeleton은 오히려 복잡해 보일 수 있다.

---

### 10.4 로딩 금지 사항

다음은 금지한다.

- 빈 화면으로 방치
- 영어 Loading만 표시
- 무한 spinner만 크게 표시
- 로딩 중 버튼 중복 클릭 허용
- 로딩 후 레이아웃이 갑자기 크게 흔들림

---

## 11. 빈 상태 규칙

### 11.1 빈 상태 목적

빈 상태는 데이터가 없을 때 사용자에게 현재 상황과 다음 행동을 알려주는 역할을 한다.

---

### 11.2 빈 상태 문구

Home:

```txt
아직 보여줄 맛집이 없어요.
조금 뒤에 다시 확인해보세요.
```

Wishlist:

```txt
아직 찜한 맛집이 없어요.
마음에 드는 맛집을 꿀단지에 담아보세요.
```

Ranking:

```txt
아직 랭킹에 올라온 맛집이 없어요.
첫 번째 꿀맛집을 추천해보세요.
```

Search/Filter:

```txt
조건에 맞는 맛집이 없어요.
다른 카테고리를 선택해보세요.
```

---

### 11.3 빈 상태 CTA

가능하면 다음 행동을 제공한다.

예시:

```txt
맛집 둘러보기
다시 불러오기
다른 카테고리 보기
```

---

### 11.4 빈 상태 금지 사항

다음은 금지한다.

- 빈 화면만 보여주기
- No Data 표시
- 사용자가 다음에 뭘 해야 하는지 모르게 하기
- 기능 없는 버튼을 제공하기

---

## 12. 에러 상태 규칙

### 12.1 에러 상태 목적

에러 상태는 문제가 발생했을 때 사용자가 상황을 이해하고 복구할 수 있게 하는 역할을 한다.

---

### 12.2 에러 문구

## 13. Toast / 피드백 규칙

### 13.1 Toast 목적

Toast 또는 짧은 피드백은 사용자의 행동 결과를 알려준다.

사용 예시:

```txt
찜 목록에 담았어요.
찜을 해제했어요.
검색 기능은 곧 추가될 예정이에요.
다시 시도해주세요.
```

---

### 13.2 Toast 사용 기준

다음 상황에서 사용할 수 있다.

- 찜 성공
- 찜 해제
- 미구현 기능 클릭
- 저장 성공
- API 실패
- 필터 결과 없음 안내

---

### 13.3 라이브러리 추가 기준

프로젝트에 이미 Toast 라이브러리가 있으면 그것을 사용한다.

없다면 Codex는 UI 리팩토링 중 Toast 라이브러리를 새로 추가하지 않는다.  
대신 간단한 inline feedback 또는 기존 알림 방식을 사용한다.

---

## 14. 애니메이션 규칙

### 14.1 애니메이션 목적

애니메이션은 사용자에게 부드러운 피드백을 주기 위한 보조 요소다.

Honeytong은 과한 애니메이션을 사용하지 않는다.

---

### 14.2 권장 애니메이션

- 버튼 tap scale
- 카드 tap scale
- 부드러운 opacity transition
- 짧은 slide/fade
- Skeleton pulse

권장 시간:

```txt
150ms ~ 250ms
```

Tailwind 예시:

```tsx
className="transition duration-200 ease-out active:scale-[0.98]"
```

Framer Motion 사용 시:

```tsx
whileTap={{ scale: 0.98 }}
transition={{ duration: 0.18 }}
```

---

### 14.3 Hover 규칙

Honeytong은 모바일 우선이므로 hover에 의존하지 않는다.

데스크톱에서는 보조 효과로만 사용한다.

예시:

```tsx
hover:bg-gray-50
```

---

### 14.4 애니메이션 금지 사항

다음은 금지한다.

- 500ms 이상의 긴 애니메이션 남발
- 화면 전체가 과하게 흔들리는 효과
- 게임처럼 화려한 랭킹 애니메이션
- hover가 없으면 기능을 알 수 없는 UI
- 애니메이션 때문에 클릭 반응이 느려지는 것

## 16. 접근성 규칙

### 16.1 기본 접근성

Codex는 UI 리팩토링 중 기본 접근성을 지킨다.

- 버튼에는 의미 있는 텍스트 또는 aria-label 제공
- 이미지에는 alt 제공
- 아이콘만 있는 버튼에는 aria-label 제공
- 텍스트 대비 유지
- 터치 영역 충분히 확보
- 키보드 포커스를 완전히 제거하지 않기

---

### 16.2 이미지 alt

맛집 이미지 alt 예시:

```tsx
alt={`${restaurantName} 대표 이미지`}
```

이미지가 장식용이면 빈 alt를 사용할 수 있다.

```tsx
alt=""
```

---

### 16.3 접근성 금지 사항

다음은 금지한다.

- 아이콘 버튼에 label 없음
- 이미지 alt 없음
- 텍스트 대비가 너무 낮음
- focus outline을 완전히 제거
- 버튼이 아닌 div에 클릭 이벤트만 붙임
- 키보드로 접근 불가능한 주요 기능

---

## 17. 미구현 기능 처리 규칙

### 17.1 미구현 기능 기준

아직 백엔드나 프론트 기능이 완성되지 않은 경우, Codex는 UI만 진짜 기능처럼 만들어서는 안 된다.

---

### 17.2 처리 방식

가능한 처리:

1. 버튼 숨김
2. 비활성 버튼
3. “곧 추가될 기능이에요.” 안내
4. TODO 주석 또는 docs/PROGRESS.md에 기록

문구 예시:

```txt
곧 추가될 기능이에요.
검색 기능은 곧 추가될 예정이에요.
이 필터는 곧 추가될 예정이에요.
```

---

### 17.3 미구현 기능 금지 사항

다음은 금지한다.

- 미구현 기능을 활성 CTA로 표시
- 클릭해도 아무 반응 없이 방치
- mock 데이터로 기능이 되는 것처럼 꾸미기
- 백엔드 API 없이 임시 성공 처리
- 사용자에게 실제 저장/추천된 것처럼 보여주기

## 20. 인터랙션 완료 체크리스트

Codex가 인터랙션을 수정했다면 다음을 확인한다.

- [ ] 카드 클릭 시 Detail로 이동하는가?
- [ ] 찜 버튼 클릭 시 Detail 이동이 발생하지 않는가?
- [ ] 찜 상태가 시각적으로 구분되는가?
- [ ] API 실패 시 안내 또는 상태 복구가 있는가?
- [ ] 하단 네비게이션 활성 상태가 맞는가?
- [ ] 필터 선택 상태가 명확한가?
- [ ] 로딩 상태가 빈 화면이 아닌가?
- [ ] 빈 상태에 안내 문구가 있는가?
- [ ] 에러 상태에 복구 버튼이 있는가?
- [ ] 버튼 터치 영역이 충분한가?
- [ ] 아이콘 버튼에 aria-label이 있는가?
- [ ] UI 문구가 한국어로 유지되는가?
- [ ] 기존 API 연결이 유지되는가?
- [ ] 기존 라우팅이 유지되는가?
- [ ] TypeScript 오류가 없는가?

## 21. Codex 작업 금지 사항

Codex는 인터랙션 리팩토링 중 다음을 하면 안 된다.

- 기존 API 호출 제거
- 기존 라우팅 경로 변경
- 찜 기능 삭제
- 상세 이동 삭제
- 하단 네비게이션 삭제
- mock 데이터 새로 만들기
- 검색/필터 미구현 기능을 실제처럼 꾸미기
- 클릭 가능한 것처럼 보이는데 클릭되지 않는 UI 만들기
- 버튼 클릭 후 아무 반응 없게 만들기
- 카드 클릭과 찜 버튼 이벤트 충돌 방치
- 영어 UI 문구로 변경
- 너무 긴 애니메이션 추가
- 사용하지 않는 라이브러리 추가
- TypeScript 오류가 있는 상태로 종료
- docs/PROGRESS.md 기록 없이 종료
